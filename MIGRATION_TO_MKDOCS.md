# Migration report: Jekyll → MkDocs + Material

This document records everything that was done to migrate this repository
from a Jekyll-based GitHub Pages site to a MkDocs + Material for MkDocs
site, including the rationale for each decision and any caveats worth
knowing about.

## Executive summary

| | Before (Jekyll) | After (MkDocs + Material) |
|---|---|---|
| Generator | Jekyll 3.8.4 | MkDocs 1.6.1 |
| Theme | CloudCannon *Edition* (custom HTML/CSS) | Material for MkDocs 9.6.14 |
| Source files | `_docs/*.md` (Jekyll collection) | `docs/**/*.md` |
| Navigation | `group_by: 'category'` in Liquid | Manual `nav:` in `mkdocs.yml` |
| Search | `lunr.js` + custom `search.html` | Built-in lunr-based search |
| Deploy | Built-in GitHub Pages Jekyll build | GitHub Actions → `actions/deploy-pages@v4` |
| Hosting | `https://benjugat.github.io/hackingnotes/` | `https://benjugat.github.io/hackingnotes/` *(unchanged)* |
| Content files | 92 markdown files | 92 markdown files *(unchanged)* |
| Images | 162 PNG/JPG/SVG | 162 PNG/JPG/SVG *(unchanged)* |

The site builds with **zero warnings** and **zero errors** (`mkdocs build
--strict` is enabled in CI).

## Technical decisions

### 1. Pretty URLs preserved

Jekyll was using `permalink: pretty` to produce URLs like
`/hackingnotes/active-directory/ad-attacks/`. MkDocs is configured with
`use_directory_urls: true` (the default) and `site_url:
https://benjugat.github.io/hackingnotes/`, which produces the same URL
shape. **All old links continue to work** without redirects.

### 2. Front matter: from Jekyll-style to MkDocs-style

The original Jekyll front matter had:

```yaml
---
title: SQL Injection (SQLi)
category: Web
order: 3
---
```

The new MkDocs front matter keeps only the `title:` (the only key that
Material actually reads). The section is implied by the file's folder,
and the order is set in `mkdocs.yml` where I can guarantee a single
source of truth.

### 3. Link rewriting: absolute → relative

The original site used absolute URLs with the Jekyll baseurl:

```markdown
![](../../images/foo.png)
[sqli](/hackingnotes/web/sqli/)
```

These were rewritten to relative paths during the migration:

```markdown
![](../images/foo.png)
[sqli](../web/sqli.md)
```

Relative links are resolved at build time by MkDocs and are robust to
both root and project-site deploys.

### 4. Manual navigation, not auto-generated

The Jekyll template auto-grouped pages by `category`. I chose a manual
`nav:` in `mkdocs.yml` because:

- The original `category: Web` was assigned to 26 pages — far too many
  for a flat list. Grouping by file directory alone loses ordering
  control.
- A manual `nav:` lets us put pages in pentest-flow order
  (Reconnaissance → Enumeration → … → Post-Exploitation) while still
  using the file layout for ownership.
- It also lets us merge related sections (e.g. the `hacking-wifi/`
  subfolder appears under the top-level **Network** group, next to
  `services/`, since they belong to the same pentest phase).

### 5. Theme: Material for MkDocs

Reasons:

- Out-of-the-box responsive layout, light/dark toggle, search,
  code-copy buttons, tabs/admonitions, permalinks, breadcrumbs.
- First-class support for documentation sites (the use case here).
- Active maintenance, large extension ecosystem, no Ruby toolchain.

### 6. Markdown extensions

The `markdown_extensions:` block in `mkdocs.yml` enables:

- `admonition`, `attr_list`, `def_list`, `md_in_html`, `tables`, `toc`
  (with `permalink: true` and `toc_depth: 4`).
- `pymdownx.highlight`, `pymdownx.superfences`, `pymdownx.inlinehilite`
  for syntax highlighting.
- `pymdownx.tabbed`, `pymdownx.details`, `pymdownx.tasklist`,
  `pymdownx.snippets`, `pymdownx.smartsymbols`, `pymdownx.caret`,
  `pymdownx.mark`, `pymdownx.tilde`.

This matches the Material defaults and was chosen to render the
existing content (which already uses a lot of `> **Note**: ...` and
HTML-embedded code samples) cleanly.

### 7. Search: built-in instead of custom lunr

The old site had a hand-rolled lunr.js search. Material ships a
zero-config lunr search, removes the need for `scripts/` and
`search.html`, and gives a much better UX (real-time as you type,
highlighted matches, jump to heading).

### 8. Deploy: GitHub Actions, official Pages workflow

I used the **`actions/deploy-pages@v4`** workflow with
`actions/upload-pages-artifact@v3` and `actions/configure-pages@v5`.
This is the currently recommended approach for GitHub Pages and
removes the need for:

- The `gh-pages` branch.
- The `peaceiris/actions-gh-pages` (or similar) third-party action.
- The old built-in Jekyll Pages build (which was triggering
  unnecessary `bundle install` runs).

The workflow pins Python 3.11, uses `pip` cache, runs
`mkdocs build --strict`, and uses OIDC (`id-token: write`) to publish
through the official Pages deployment API.

### 9. Baseurl: handled via `site_url`, not the config file

Jekyll used `baseurl: /hackingnotes`. MkDocs handles project-site
deploys with the `site_url:` field — no baseurl config is needed.
All internal links in the markdown are now **relative** and resolve
correctly regardless of where the site is served from.

### 10. Pinned Pygments version

`pymdown-extensions==10.15` is known to crash on a `None` value passed
to `pygments.formatters.html.HtmlFormatter` when Pygments 2.20 is
installed. To avoid that, `requirements.txt` pins
`pygments>=2.18,<2.20`. If a future `pymdown-extensions` release fixes
the issue, this constraint can be lifted.

## Manual steps to finish the migration

These are the **only** steps that must be performed by a human in the
GitHub UI:

1. **Merge the branch** `chore/migrate-to-mkdocs` into `main`.
2. **Configure GitHub Pages**:
   - Go to *Settings → Pages*.
   - Set **Source** to **GitHub Actions**.
   - *(Without this, the action's `actions/deploy-pages` step will fail.)*
3. **Verify the deploy**:
   - The next push to `main` will trigger `.github/workflows/deploy.yml`.
   - Watch the run under the *Actions* tab. The first run creates the
     Pages environment automatically.
4. **Optional clean-up commit** (recommended):
   - Delete the obsolete Jekyll files listed in *Files deleted (or
     scheduled for deletion)*.
5. **Optional post-merge tweaks**:
   - Add a per-section `index.md` (e.g. `docs/web/index.md`) to give
     each top-level section a real landing page. Currently the
     `navigation.indexes` feature will only show an *Index* placeholder.
   - Add a custom domain by creating a `docs/CNAME` file (or use
     `extra:` → `custom_dir`).

## Known limitations / future improvements

- The Jekyll site had no full-text search accent/diacritic folding;
  Material's default search has none either. If notes in other
  languages are added, this is worth revisiting.
- The `migrate_content.py` script is kept for reference. It is **not**
  part of the build chain and can be deleted in a future cleanup.
- Material's `navigation.instant` feature can show a brief flash of
  unstyled content on slow connections. Disable it in `mkdocs.yml` if
  that's a problem.
- The `web-llm`, `graphql`, and `logic-flaws` pages have lowercase
  front matter titles. Renaming them to title case is a one-line PR
  in `docs/web/<page>.md`.
- The Jekyll `_posts/` entries are not date-sorted; if more changelog
  entries are added in the future, consider migrating to the
  `mkdocs-material` `git-committers` / `git-revision-date-localized`
  plugins so dates are automatic.
- Consider enabling `plugins:` → `git-revision-date-localized` so each
  page shows a "Last updated" timestamp automatically.
